# P2: Builder — PPT Generation & Component Population

## Role

Builder turns the layout spec into a real PPT file. Two generation modes:

| Mode | Method | Best For |
|------|--------|----------|
| **SVG-first** (DEFAULT) | Hand-write SVG → ppt-master `svg_to_pptx.py` | All figures. LLM-native precision, browser preview, pixel-perfect layout. |
| python-pptx | Direct python-pptx API calls | Fallback when ppt-master unavailable. Requires Precision Discipline (named constants, group declarations, typography-first). |

## Input

- `figures/<name>/figure-layout.md` (from P1 — narrative layout spec, MUST include chosen Layout Pattern)
- `figures/<name>/figure-spec-lock.md` (from P1 — machine-readable execution contract)
- `figures/paper-style-lock.md` (paper-wide style lock — shared typography/colors/spacing)
- Template PPT file (matched in P0)
- User feedback from P1 confirmation

## Process (SVG-first — DEFAULT)

### Step 0: Read Contracts

```
Read figures/<name>/figure-layout.md   → extract Layout Pattern, column budgets, group inventory
Read figures/<name>/figure-spec-lock.md → extract typography roles, colors, shape vocabulary, spacing
Read figures/paper-style-lock.md       → extract shared constraints
```

### Step 1: Set Up SVG Canvas

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 1780 1100"
     font-family="Times New Roman, Times, serif">
```

| Column Width | viewBox | px Equivalent |
|-------------|---------|---------------|
| Double (17.8cm) | `0 0 1780 1100` | 100 px/cm |
| Single (8.9cm) | `0 0 890 1100` | 100 px/cm |

**Hard rule**: viewBox dimensions MUST match the spec_lock canvas values exactly.

### Step 2: Write SVG Layer-by-Layer

SVG DOM order = z-order (earlier elements are behind later elements).

```
L1: <defs> markers, then <line> connectors (behind everything)
L2: <rect> region backgrounds, container panels
L3: Sub-components — <rect rx>, <ellipse>, preset <g> fragments
L4: Highlights — preset star <g>, reading markers
L5: <text> elements (always last — always on top)
```

**Hard rule — markers**: Every `<line marker-end="url(#m)">` must use a marker whose `fill` matches the line's `stroke` EXACTLY. Create one marker per stroke color.

**Hard rule — presets**: Native PPT shapes (CHEVRON, CAN, STAR5, etc.) MUST be generated by `preset_shape_svg.py`. Never hand-edit the path data inside a preset `<g>`. The validator checks hashes. If coordinates change, re-run the tool.

**Hard rule — text positioning**: Every `<text>` element gets exact `x`, `y` coordinates. Use `text-anchor="middle"` for centered labels, `text-anchor="start"` for left-aligned. No auto-sizing — you control exactly where text lands.

### Step 3: Browser Preview

Open the SVG file in a browser. Verify:
- All elements visible in correct positions
- No text cut off or overlapping
- Layer order correct (connectors behind shapes, text on top)
- Tree structure recognizable (root → branches → leaves)
- Column proportions match the Layout Pattern budget

Fix issues in SVG BEFORE converting to PPTX.

### Step 4: Convert to PPTX

```bash
# Copy SVG to project
cp fig-01-v{N}.svg svg-project/svg_output/P01_figure.svg

# Convert
source activate paper-plot
python skills/ppt-master/scripts/svg_to_pptx.py svg-project \
  -o figure-v{N}.pptx
```

**Troubleshooting**:
- "marker fill does not match stroke" → Fix marker color
- "preset path differs from registry" → Re-run `preset_shape_svg.py`
- "viewBox does not match locked canvas" → Fix viewBox dimensions
- "missing colors" → Check spec_lock.md has all required roles

### 0. Precision Discipline — Read Specs First (MANDATORY)

Before writing a single line of shape code, read the execution contracts:

```
Read figures/<name>/figure-spec-lock.md      # Per-figure: all numeric & semantic values
Read figures/paper-style-lock.md             # Paper-wide: shared typography/colors/spacing
```

Every value used in generation MUST come from these files. No inline literals for font sizes, colors, coordinates, or spacing. No values from memory.

### 1. Typography-First Build (MANDATORY — 7 steps)

**Precision Discipline:** All text dimensions are calculated BEFORE containers are sized. Never shrink text to fit a container — expand the container to fit text.

#### Step A: Read Spec Locks

Extract all constants from `figure-spec-lock.md` and `paper-style-lock.md`:
- `typography_roles` → font sizes per role
- `color_roles` → fill/stroke colors per semantic role
- `shape_vocabulary` → approved MSO_SHAPE types
- `spacing_constants` → gaps, margins, padding
- `layer_inventory` → complete element list per layer
- `groups` → group declarations with bbox and elements

#### Step B: Build Text Inventory

List ALL text elements with their typography role BEFORE placing any shape:

```python
# === TEXT INVENTORY (before any shape code) ===
TEXT_INVENTORY = [
    # (element_id, text_content, role, group, approx_chars)
    ("title_hero", "Collaborative Skill Evolution", "figure_title", "hero", 30),
    ("label_cloud", "CLOUD BRAIN", "section_title", "cloud-header", 11),
    ("desc_analyzer", "Frozen LLM performs...", "sub_label", "cloud-analyzer", 25),
    # ... ALL text elements must be listed here
]
```

For each text element, estimate dimensions:
- Single-line width ≈ char_count × font_size_pt × 0.06 cm (Times New Roman)
- Multi-line height ≈ line_count × font_size_pt × 0.045 cm

#### Step C: Calculate Text Dimensions

Use the estimation rule:

```python
def estimate_text_size(text, font_size_pt, max_width_cm=None):
    """Estimate text dimensions. Returns (width_cm, height_cm)."""
    char_width = font_size_pt * 0.06       # approximate cm per char (TNR)
    line_height = font_size_pt * 0.045     # approximate cm per line
    total_width = len(text) * char_width
    if max_width_cm and total_width > max_width_cm:
        lines = math.ceil(total_width / max_width_cm)
        return (max_width_cm, lines * line_height)
    return (total_width, line_height)
```

**Hard rule**: Minimum text box width ≥ width of the longest single word + 0.2cm.

#### Step D: Derive Container Sizes FROM Text

For each container, calculate size based on its text content:
- Container width = max(all_text_widths_in_container) + 2 × container_padding
- Container height = sum(all_text_heights) + gaps + 2 × container_padding

```python
# Container sizing example — container expands to fit text
leaf_texts = [t for t in TEXT_INVENTORY if t[3] == "skill-tree-leaves"]
max_text_w = max(estimate_text_size(t[1], ROLE_SIZES[t[2]])[0] for t in leaf_texts)
max_text_h = max(estimate_text_size(t[1], ROLE_SIZES[t[2]])[1] for t in leaf_texts)
leaf_w = max_text_w + 2 * GAP['container_padding']
leaf_h = max_text_h + 2 * GAP['container_padding']
```

**Hard rule**: Never reduce font size to fit a container. Enlarge the container instead.

#### Step E: Compute Layout from Named Constants

With container sizes known, calculate absolute positions using named layout constants. All coordinates derive from the spec lock's region definitions and spacing constants:

```python
# === LAYOUT CONSTANTS (from figure-spec-lock) ===
W, H = 17.8, 11.0
ML, MR, MT, MB = 0.4, 0.4, 0.5, 0.5

REGION = {
    'cloud': {'x': ML, 'y': 0.5, 'w': 4.8, 'h': 10.0},
    'hero':  {'x': 5.5, 'y': 0.6, 'w': 6.8, 'h': 9.0},
    'edge':  {'x': 12.6, 'y': 0.5, 'w': 4.8, 'h': 10.0},
}

SZ = {'title': 10, 'section': 9, 'module': 7, 'body': 6.5, 'annotation': 6, 'detail': 5.5}
GAP = {'intra': 0.15, 'inter': 0.3, 'region': 0.4, 'pad': 0.2}
```

**Forbidden**: Raw float literals in shape coordinates. Every coordinate must be derived from named constants or a documented computation.

#### Step F: Place Shapes Layer-by-Layer, Group-by-Group

Draw ALL elements of one layer before moving to the next (see Drawing Order below). Within each layer, process by group — complete one group (with self-check) before moving to the next.

#### Step G: Verify All Text Fits

After all elements are placed, check every text container:
- Text width ≤ container width − 2 × text_margin
- Longest word fits in one unwrapped line
- No text overflow or clipping
- Luminance-aware text color is correct on every filled shape

Re-check after any coordinate or font adjustment during iteration.

### Shape Type Mapping

**Minimum 3 distinct MSO_SHAPE types per figure.** A figure with only rectangles is not acceptable.

| Layout Element | PPT Shape Type | When to Use |
|---------------|----------------|-------------|
| Component box | `MSO_SHAPE.ROUNDED_RECTANGLE` | System modules, functions |
| Status node / state | `MSO_SHAPE.OVAL` | Lifecycle states, tree nodes, endpoints |
| Decision point | `MSO_SHAPE.DIAMOND` | Branching in flowcharts |
| Process step / flow | `MSO_SHAPE.CHEVRON` | Pipeline stages, sequential flow |
| Data store | `MSO_SHAPE.CAN` | Databases, file storage, pools |
| Highlight / special | `MSO_SHAPE.STAR_5_POINT` or `STAR_4_POINT` | Critical insights, key contributions (≤2 uses) |
| Cloud service | `MSO_SHAPE.CLOUD` | Remote/cloud-hosted components |
| Document / file | `MSO_SHAPE.FOLDED_CORNER` | Files, documents, reports |
| Tag / badge (pill) | `MSO_SHAPE.ROUNDED_RECTANGLE` + high radius | Short labels, categories |
| Connector | `MSO_CONNECTOR.STRAIGHT` | Data/control flow |
| Text label | `MSO_SHAPE.TEXT_BOX` | Annotations, titles |
| Table | `ppt.table` | Comparisons, results |
| Group boundary | `MSO_SHAPE.RECTANGLE` (dashed, no fill) | Layer/group outlines (use sparingly — whitespace is preferred) |

### Group Declaration (MANDATORY — before any shape code)

Every element in the figure belongs to exactly one named group. No orphan elements. Declare ALL groups before writing shape code:

```python
# ============ GROUP DECLARATIONS ============
GROUPS = {
    "cloud-analyzer": {
        "layer": "L3",
        "parent": "cloud-region",
        "structure": "pipeline",
        "bbox": (0.4, 0.5, 4.8, 2.5),
        "elements": ["analyzer_bg", "stage_1", "stage_2", "stage_3"],
    },
    "skill-tree": {
        "layer": "L3",
        "parent": "cloud-region",
        "structure": "tree",
        "bbox": (0.4, 3.3, 4.8, 4.5),
        "elements": ["tree_root", "branch_1", "branch_2", "branch_3",
                      "leaf_01", "leaf_02", "leaf_03", "leaf_04", "leaf_05",
                      "leaf_06", "leaf_07", "leaf_08", "leaf_09"],
    },
    # ... every group declared
}
```

Within each layer, wrap each group's code in an ASCII box comment with self-check:

```python
# ============================================================
# GROUP: skill_tree | L3 | cloud-region | tree structure
# Elements: 1 root oval + 3 branch lines + 9 leaf nodes
# BBox: (0.4, 3.3, 4.8, 4.5)
# ============================================================

# Root node
CIR(slide, REGION['cloud']['x'] + 2.4, 4.55, 0.38, ...)

# Branch nodes
for i, cx in enumerate([1.1, 2.8, 4.5]):
    CIR(slide, REGION['cloud']['x'] + cx, 5.85, 0.25, ...)

# Leaf nodes
for i, (label, bx) in enumerate(leaf_data):
    R(slide, REGION['cloud']['x'] + bx, 6.35, LEAF_W, LEAF_H, ...)

# Self-check:
# [PASS] 13 elements in bbox (tolerance 0.1cm)
# [PASS] All text fits inside containers
# [PASS] No internal overlaps (min intra-group gap: 0.15cm)
# ============================================================
```

**Per-group self-check (after each group completes):**
- [ ] Element count matches group declaration
- [ ] All elements within declared bbox (tolerance: 0.1cm)
- [ ] All text fits in containers (no overflow)
- [ ] No internal overlaps
- [ ] Connectors anchored to shape edges (not floating)

### Drawing Order (MANDATORY — Layer-by-Layer, Group-by-Group)

The drawing order directly determines z-order. **Draw ALL elements of one layer before moving to the next.** Never interleave layers.

```
LAYER 1 (first):  ALL connectors & arrows → reorder_connectors_behind()
LAYER 2:           ALL module containers, backgrounds, panels
LAYER 3:           ALL sub-components, cards, circles, icons, CAN, CHEVRON
LAYER 4:           ALL highlights, stars, accent badges, reading markers ①②③
LAYER 5 (last):    ALL text boxes, labels, annotations → always on top
```

**Why this order matters:**
- If you draw a connector (L1) after a container (L2), the connector renders ON TOP of the container → visual bug
- If you draw a text box (L5) before a highlight (L4), the text can be obscured by the highlight
- Interleaving layers creates accidental occlusion that is hard to debug

**Code organization rule:** Your generation script MUST group code by layer, not by region. Structure your code as:

```python
# === L1: CONNECTORS ===
# (all arrows, all regions)

# === L2: CONTAINERS ===
# (all module backgrounds, region tints)

# === L3: SUB-COMPONENTS ===
# (all cards, circles, icons)

# === L4: HIGHLIGHTS ===
# (all stars, badges, markers)

# === L5: TEXT ===
# (all text boxes, labels)
```

Do NOT organize by region (don't do: all of Region-A L1-L5, then all of Region-B L1-L5, then all of Region-C L1-L5). Layer-first, group-second, region-third.

**Group discipline**: Within each layer, complete one group (declare → draw → self-check) before moving to the next group. Never interleave groups.

### Connector Routing

Follow the routing discipline from style-guide:
- ONE connector per logical connection (no chained straight segments)
- Connectors begin/end at shape edges, not arbitrary coordinates
- Connector labels positioned near the line midpoint
- Bidirectional pairs: parallel, spaced 0.3-0.5cm, clear directional labels
- All connectors behind all shapes (z-order enforced)

### python-pptx Key Operations

```python
from pptx import Presentation
from pptx.util import Inches, Cm, Pt, Emu
from pptx.enum.shapes import MSO_SHAPE, MSO_CONNECTOR_TYPE
from pptx.dml.color import RGBColor
from pptx.oxml.ns import qn

# Clone slide dimensions from template
prs = Presentation(template_path)
slide = prs.slides[slide_index]

# Add shape
shape = slide.shapes.add_shape(
    MSO_SHAPE.ROUNDED_RECTANGLE,
    Cm(x), Cm(y), Cm(w), Cm(h)
)
shape.text = "Component Name"

# Add connector
connector = slide.shapes.add_connector(
    MSO_CONNECTOR_TYPE.STRAIGHT,
    Cm(x1), Cm(y1), Cm(x2), Cm(y2)
)

# Adjust slide height dynamically
slide_height = max(Cm(4), calculated_height)
prs.slide_height = slide_height
```

### Slide Height Calculation

Fixed width + flexible height: sum all vertical regions + gaps + margins. Minimum 4cm (template default).

## Failure Modes — Watch For These

### Text & Font
1. **Text overflow after font change:** When Times New Roman isn't available and system substitutes another serif font, text may overflow its bounding box. Check each box's text fits. Fix: reduce font size by 0.5-1pt or enlarge box.
2. **Placeholder text in output:** "TODO", "???", "Label", "Component" left in final shapes. Check every shape has real text before P3 gate.
3. **Single-line truncation:** Long component names get clipped. Enable `word_wrap=True` on all text frames.
4. **Wrong font weight in sub-labels:** Sub-text paragraphs inheriting bold from main text. Explicitly set `run.font.bold = False` on sub-label runs.

### Layout & Positioning
5. **Nested elements drawn in global coordinates:** When a component should be inside a panel, giving it absolute (x, y) instead of panel-relative coordinates. This causes drift when the panel moves. Use panel-local coordinates when panels exist.
6. **Cross-panel overlap:** Children drawn outside their parent panel bounds, or adjacent panels colliding. Check bounding boxes before export.
7. **Arrows through text boxes:** Connector lines passing through unrelated component labels. Re-route connectors or adjust label positions.
8. **Lower rows drifting:** When font changes or shape resizes cascade downward, pushing lower rows out of alignment. Re-check all positions after any font/shape size change.

### Style & Color
9. **Uniform palette when color distinguishes submodules:** Using the same fill for all components when the figure's semantics rely on color grouping. Respect the role mapping from P0.
10. **Missing dashed distinction:** Using solid lines for feedback/optional flows when dashed was specified. Check connector semantics against figure-intent.
11. **Equations or special symbols as images:** Pasting screenshots instead of using Unicode or native text approximations. Keep text editable.

### Export & Save
12. **Exporting before saving:** Running PNG export on an unsaved in-memory PPT. Always `prs.save()` before export.
13. **Stale exports:** Exporting from an old version of the file when multiple v{N}.pptx exist. Verify the file path being exported matches the latest save.
14. **Template reference images in output:** Residual PNG/JPG in the `.pptx` package from template cloning. The `clean_slide()` function prevents this, but verify with `InspectPackage`-equivalent: check for unexpected media entries. (`.pptx` is a ZIP; `visio/media/` equivalent is `ppt/media/`)

### Text-Shape Binding
15. **Floating labels:** Text boxes placed at absolute coordinates without verifying proximity to the shape they describe. A label 0.5cm from shape A and 0.5cm from shape B is ambiguous. Check: label-to-own-shape distance ≤ 50% of label-to-nearest-other-shape distance.
16. **Text box too narrow:** Box width cannot fit the longest word at the given font size. At 7pt Times New Roman, "Internalized" ≈ 1.2cm. A 0.8cm-wide box will break mid-word. Minimum width rule: measure the longest word; box must be ≥ that width + 0.2cm.
17. **Text box too wide:** Multi-word labels that sprawl as a single line across the canvas. A 5+ word label that fits on one line means the box is too wide. Adjust width so text wraps to 2-3 lines.
18. **Text color doesn't contrast fill:** White text on light fill (invisible) or dark text on dark fill (unreadable). Apply `is_dark()` to every filled shape's fill color before setting text color.

### Connector Routing
19. **Chained straight segments:** Using 2-3 separate straight connectors to simulate one routed line. Joints between segments are never perfectly aligned → visible gaps. Use ONE connector per logical connection; route through shape edges or dedicated channels.
20. **Connectors not behind shapes:** A connector line visible ON TOP of a component box or label. The `reorder_connectors_behind()` call was missed or failed. Verify z-order before every save.
21. **Connector endpoints floating:** A connector that starts/ends at arbitrary (x, y) rather than at a shape edge. The line appears disconnected from both source and target. Calculate endpoints to touch shape boundaries.

### Spec-Lock & Precision Discipline
22. **Raw float literals in coordinates:** A shape call like `R(slide, 0.4, 0.5, 4.8, 2.5)` with no named constant. All coordinates must derive from REGION dict, margin constants, or computed variables. Inline floats are FORBIDDEN except for trivial loop counters or accumulators. Fix: replace `0.4` with `REGION['cloud']['x']` or `ML`.
23. **Inline font sizes:** A `TB()` call with a literal number like `size=7` instead of `size=SZ['module']`. All font sizes must reference the SZ dict, which is populated from figure-spec-lock.md. Fix: declare SZ dict at top, use `SZ['module']` everywhere.
24. **Orphan element (no group):** A shape or text box created outside any declared group. Every element must appear in exactly one GROUP declaration and its code must be wrapped in the corresponding group comment block. Fix: find which group this element belongs to, add it to the group's elements list.
25. **Container-too-small (text-first violation):** A container sized before measuring its text content, resulting in overflow or squeezed text. Container sizes must be DERIVED from text dimensions (Step D), not guessed. Fix: measure text first, enlarge container to fit.
26. **Spec-lock value drift:** Using a color, font size, or spacing value not present in figure-spec-lock.md. P2 must not invent values. If a genuinely new value is needed: stop, update the spec-lock, then continue. Fix: read spec-lock, use only listed values.
